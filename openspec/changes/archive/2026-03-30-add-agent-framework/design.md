# Agent 框架技术设计

## Context

### 背景

Chroplus 是一个 Manifest V3 Chrome 扩展，已实现基于 Gemini API 的 AI 聊天侧边栏。现在需要扩展为 Agent 架构，支持工具调用和多轮自主执行。

### 约束

- **无外部依赖**: 不引入 LangChain.js 等重型框架，保持轻量
- **Gemini Function Calling**: 使用 Gemini 原生支持的 Function Calling API
- **Side Panel 执行**: Agent 循环在 Side Panel 中运行，避免 Service Worker 被终止
- **代码可读性**: 单个文件不超过 600 行，模块化设计
- **向后兼容**: 不破坏现有 AI 聊天功能

### 利益相关者

- 用户：需要 Agent 执行复杂任务（如整理书签）
- 开发者：需要清晰易扩展的 Agent 框架

## Goals / Non-Goals

**Goals:**

- 搭建 Agent 核心框架，支持多轮工具调用循环
- 实现工具注册机制，支持声明式定义工具
- 集成 Gemini Function Calling API
- 代码模块化，文件职责清晰

**Non-Goals:**

- 不实现具体业务工具（书签整理等）
- 不支持多 Agent 协作
- 不支持流式输出（Function Calling 场景下暂不需要）
- 不实现工具执行超时控制（MVP 阶段）

## Decisions

### 1. Agent 循环执行位置

**决定**: 在 Side Panel 中执行 Agent 循环

**原因**:

- Service Worker 会在空闲 30 秒后被终止，长任务可能中断
- Side Panel 只要打开就保持活跃
- 工具调用在 Side Panel 中执行更直观

**替代方案**:

- ~~在 Background Service Worker 中执行~~: 可能被终止，不适合长任务
- ~~混合模式~~: 增加复杂度，MVP 不需要

### 2. 模块拆分策略

**决定**: 按职责拆分为多个小文件

**文件结构**:

```
agent/
  ├── types.js          # 类型定义、常量 (~100 行)
  ├── tool-registry.js  # 工具注册中心 (~200 行)
  └── executor.js       # Agent 循环执行器 (~300 行)

lib/
  └── gemini-tools.js   # Gemini Function Calling 格式转换 (~150 行)

background.js           # GeminiProvider 扩展 (~400 行，已有部分)
sidepanel.js            # 主入口，组装各模块 (~300 行)
```

**原因**:

- 单一职责，易于理解和测试
- 方便后续扩展（如添加新工具）
- 符合 proposal 中"单文件不超过 600 行"的要求

### 4. 工具注册设计

**决定**: 声明式注册，支持 JSON Schema

**工具定义格式**:

```javascript
// 示例：一个简单的工具定义
const tool = {
  name: 'get_current_time',
  description: '获取当前时间',
  parameters: {
    type: 'object',
    properties: {
      timezone: {
        type: 'string',
        description: '时区，如 Asia/Shanghai'
      }
    }
  },
  execute: async (args) => {
    return { time: new Date().toISOString() };
  }
};
```

**原因**:

- JSON Schema 是 Gemini Function Calling 的标准格式
- 声明式定义便于生成 Gemini 需要的 `functionDeclarations`
- 执行函数分离，便于测试和扩展

### 5. 与现有聊天功能的兼容

**决定**: 保持现有 `chat` API 不变，新增 `chatWithTools` 方法

```javascript
// GeminiProvider 接口
class GeminiProvider {
  // 已有：普通对话
  async chat(messages, onChunk, apiKey) { ... }

  // 新增：支持工具的对话
  async chatWithTools(messages, tools, apiKey) {
    // 返回 GeminiFunctionResponse
    // 可能是文本或 FunctionCall
  }
}
```

**原因**:

- 向后兼容，不影响现有聊天功能
- 职责分离，普通对话不需要处理工具逻辑

## Risks / Trade-offs

### 风险 1: Agent 无限循环

**风险**: 工具调用可能陷入无限循环

**缓解**:

- 设置最大循环次数 (10 次)
- 记录每次工具调用，便于调试

### 风险 2: 网络错误中断

**风险**: 网络不稳定导致 Agent 执行中断

**缓解**:

- MVP 阶段不实现重试，显示错误信息
- 后续可添加断点续执行能力

### 风险 3: 工具执行异常

**风险**: 工具执行可能抛出未捕获异常

**缓解**:

- ToolRegistry 包装执行函数，统一捕获异常
- 将异常信息返回给 Agent，让它决定如何处理

## Migration Plan

### 部署步骤

1. 新增 agent/ 目录和相关文件
2. 扩展 GeminiProvider，添加 `chatWithTools` 方法
3. 修改 sidepanel.js，集成 Agent 框架
4. 测试 Agent 循环和工具调用

### 回滚策略

- 新功能独立，不影响现有聊天
- 如有问题，可禁用 Agent 功能，回退到普通聊天
